# Critique Protocol

## Purpose

Use this protocol after building to catch generic output and weak craft decisions before presenting.

## Run Order

1. Swap test
2. Squint test
3. Signature test
4. Token test

## 1) Swap Test

Mentally substitute:

- Primary typeface -> default dashboard typeface
- Layout structure -> common sidebar + grid template

If result feels mostly unchanged, design defaulted. Identify which decisions were non-choices and rewrite them.

## 2) Squint Test

Blur visual detail and inspect hierarchy:

- Can section structure still be read?
- Are emphasis points intentional?
- Do borders/surfaces shout?

If noisy or flat, retune contrast, spacing, and surface deltas.

## 3) Signature Test

Point to 5 concrete places where signature appears:

- A repeated structural motif
- A distinctive information framing
- A unique control/data interaction
- A specific typographic rhythm
- A product-world metaphor translated into UI

If you cannot find 5, signature is too weak.

## 4) Token Test

Read token names as language. If names could belong to any template, revise toward product-world vocabulary.

## Rebuild Rule

If any test fails, revise before presenting. Do not ship first draft when critique reveals defaulting.
